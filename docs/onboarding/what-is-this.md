# 이게 뭐야? (3분)

> **유형**: Explanation · **독자**: Level 0 · **읽는 시간**: ~3분

이 레포 링크를 누군가에게 막 받아 들고 "그래서 이게 대체 뭔데?" 싶을 때 읽는 글이에요. 코드를 한 줄도 안 봐도, 클론조차 안 했어도 괜찮아요. 3분이면 정체가 잡히도록 옆에서 한 단계씩 안내할게요. 처음 보는 용어가 나오면 그 자리에서 한 줄로 풀거나 [`용어 사전`](../reference/glossary.md) 으로 바로 넘어갈 수 있게 링크를 달아 뒀어요.

## 한 문장으로

한 명이 여러 개의 모바일 앱을 동시에 운영하기 위한 [`Spring Boot`](../reference/glossary.md#프레임워크--빌드) 백엔드 템플릿 레포예요.

[Spring Boot](../reference/glossary.md#프레임워크--빌드) 는 Java 로 웹 서버를 만드는 프레임워크예요. 여기에 "여러 앱을 한 명이" 라는 조건이 붙으면서 보통의 백엔드 예제와는 다른 모양이 됐어요. 그 모양이 무엇이고 왜 그렇게 됐는지를 지금부터 풀어 볼게요.

## 조금 더 풀어서

모바일 앱을 여러 개 만들고 싶은데, 각 앱마다 백엔드를 따로 세우면 관리가 금세 불가능해져요. 이 레포는 한 번 셋업해 두면 새 앱을 추가할 때 드는 비용이 거의 0 에 가깝도록 설계됐어요.

비유하자면 공장 하나로 제품 여러 개를 찍어내는 구조예요.

가계부 앱, 운동 기록 앱, 포트폴리오 트래커 앱을 떠올려 보세요. 셋은 전혀 다른 도메인이지만, 공통으로 필요한 건 똑같아요. 이메일 가입, 소셜 로그인, 푸시 알림, 파일 업로드, 그리고 앱마다 분리된 데이터베이스. 이 공통 부분을 하나의 실행 파일 안에 묶어 두고, 각 앱의 고유한 도메인만 따로 꽂는 게 이 템플릿의 핵심이에요.

## 어떤 문제를 풀었나

### 문제 1 — 혼자서 앱 여러 개를 운영하면 결국 뭔가를 포기하게 돼요

앱 A, B, C 를 각각 별도 서버에 올리면 서버 세 대를 동시에 모니터링하고, 업데이트하고, 보안 패치해야 해요. 한 명이 감당하기 어려운 양이에요. 그렇다고 [마이크로서비스](../reference/glossary.md#아키텍처-용어) 로 더 잘게 쪼개면 쿠버네티스나 분산 추적 같은 운영 도구가 줄줄이 따라붙어요. 이것도 혼자서는 벅차요. 그러다 보면 "그냥 앱 A 만 하자" 며 여러 앱을 찍어내겠다는 처음의 계획 자체를 접게 돼요.

이 템플릿의 해법은 [모듈러 모놀리스](../reference/glossary.md#아키텍처-용어) 예요. 하나의 [JVM](../reference/glossary.md#프레임워크--빌드) 프로세스 안에서 여러 앱의 도메인이 함께 살아요. 서버는 한 대, 앱은 여러 개라는 뜻이에요. 마이크로서비스의 복잡함 없이 그 이점의 일부만 취하는 절충이에요 ([`ADR-001 · 모듈러 모놀리스`](../philosophy/adr-001-modular-monolith.md)).

### 문제 2 — 새 앱을 추가할 때마다 똑같은 셋업을 반복하게 돼요

이메일 인증, 토큰 발급, 소셜 로그인 같은 로직은 앱이 달라도 거의 똑같아요. 그런데 매번 복사해 붙이면 나중에 한 군데를 고칠 때 여러 곳을 똑같이 고쳐야 하는 짐이 생겨요.

이 템플릿은 공통 로직을 `core/` 모듈로 한 번만 추출해 모든 앱이 함께 써요. 데이터는 앱마다 별도 [schema](../reference/glossary.md#데이터베이스) 로 완전히 갈라놓아 서로 섞이지 않고요. 새 앱이 필요하면 명령 한 줄이면 돼요.

```bash
./tools/new-app/new-app.sh sumtally
```

이 한 줄이 모듈 골격, 데이터베이스 schema, 공통 마이그레이션, 설정까지 한꺼번에 만들어 줘요.

### 문제 3 — 템플릿을 복사한 뒤 원본의 개선을 받아 오기 어려워요

보통 원본 레포를 [fork](../reference/glossary.md#개발-프로세스) 해서 시작하면, 나중에 원본에 좋은 기능이 추가됐을 때 그걸 내 레포로 가져오는 일이 골치 아파져요. 충돌을 일일이 풀어야 하거든요.

이 템플릿은 GitHub 의 ["Use this template" 패턴](../reference/glossary.md#이-레포-고유-용어) 으로 분기하고, 모든 커밋이 [Conventional Commits](../reference/glossary.md#개발-프로세스) 형식을 따르도록 강제해요. 커밋 메시지가 기계가 읽을 수 있는 형식이라, 공통 개선 커밋만 골라 내 레포로 [cherry-pick](../reference/glossary.md#개발-프로세스) 해 옮길 수 있어요 ([`ADR-002 · "Use this template" 패턴`](../philosophy/adr-002-use-this-template.md)).

## 당신한테 맞을 수 있는 경우

다음에 여러 개 해당할수록 잘 찾아온 거예요.

- 모바일 앱을 혼자 또는 두세 명 규모로 여러 개 만들고 싶어요.
- 각 앱의 도메인은 다르지만 인증·푸시·결제 같은 공통 기능은 재사용하고 싶어요.
- Spring Boot 에 이미 익숙하거나, 이참에 배우고 싶어요.
- Supabase·Firebase·Resend 같은 관리형 서비스를 끼워 운영 부담을 줄이고 싶어요.
- 데이터베이스는 [PostgreSQL](../reference/glossary.md#데이터베이스) 을 써도 괜찮아요.

## 당신한테 안 맞을 수 있는 경우

반대로 다음이라면 다른 선택이 더 나을 수 있어요.

- 팀이 열 명 이상이고 앱마다 전담자가 있어요. 그렇다면 마이크로서비스나 별도 레포가 더 어울려요.
- Node.js·Go·Python 처럼 다른 언어 생태계를 원해요. 이 레포는 Java 전용이에요.
- MongoDB 같은 비관계형 데이터베이스가 꼭 필요해요. 이 레포는 PostgreSQL 을 전제로 설계됐어요.
- 앱이 하나뿐이고 당분간 늘릴 계획도 없어요. 그렇다면 이 구조는 과해요. 평범한 Spring Boot 튜토리얼이 더 빨라요.
- 99.99% 가용성이나 멀티 리전, 초당 수만 요청 같은 대규모 스케일이 필요해요. 이 템플릿은 인디 규모를 전제로 해요 ([`ADR-007 · 솔로 친화적 운영`](../philosophy/adr-007-solo-friendly-operations.md)).

## 숫자로 보기

| 항목 | 값 | 의미 |
|---|---|---|
| 앱 추가 시간 | 약 1~5분 | `new-app.sh <slug>` 한 줄로 끝 |
| 공통 코드 재사용률 | 거의 100% | 인증·푸시·스토리지·이메일을 전부 공유 |
| 배포 단위 | JAR 한 개 | 앱이 열 개여도 배포는 한 번 |
| 외부 월 비용 (인디 규모) | $50~$100 | Supabase·Resend·FCM·R2·도메인 합산 |
| ADR 문서 수 | 39개 | 주요 설계 결정의 근거 기록 |
| ArchUnit 규칙 수 | 22개 (r12 예약, 활성 21) | 모듈 경계 위반은 빌드 시점에 실패 |

[JAR](../reference/glossary.md#프레임워크--빌드) 은 컴파일된 Java 코드를 한 덩어리로 묶은 실행 파일이에요. 이 템플릿은 앱이 몇 개든 이 JAR 하나로 배포해요. [ADR](../reference/glossary.md#아키텍처-용어) 은 "왜 이렇게 설계했는가" 를 기록한 결정 카드이고, [ArchUnit](../reference/glossary.md#라이브러리--sdk) 은 모듈 사이 경계 규칙을 코드로 검사하는 도구예요. 둘 다 뒤에서 다시 만나요.

## 어떤 기술을 쓰나

전체 기술 스택을 가볍게 훑어 두면 다음 문서가 한결 수월해요. 지금은 외워야 할 목록이 아니라 "이런 걸 쓰는구나" 정도면 충분해요.

| 영역 | 기술 |
|---|---|
| 언어·프레임워크 | Java 21, Spring Boot 3.5 |
| 빌드 | Gradle 멀티모듈 |
| 데이터베이스 | PostgreSQL 16 (로컬은 Docker, 운영은 Supabase) |
| 인증 | JWT (HS256) + Spring Security |
| 이메일 | Resend |
| 푸시 | Firebase Cloud Messaging |
| 스토리지 | MinIO 또는 Cloudflare R2 |
| 배포 | Docker + Kamal blue/green (맥미니 홈서버 또는 클라우드 VM) |
| 관측성 | Prometheus + Loki + Grafana |

## 한 줄 요약
