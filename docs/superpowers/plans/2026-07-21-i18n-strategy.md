**Goal:** 유저 대면 문구(에러/push/이메일)의 다국어 계약을 확정 — **서버는 `code` 만 계약, 클라가 code→arb 번역** 을 기본으로 하고, 서버 `message` 는 디버그·fallback 용으로 격하한다.

**Architecture:** 기존 `ApiError(code, message, details)` wire 포맷은 **무변경**(구버전 앱 호환). 의미론만 재정의: 클라 표시는 `code→arb 매핑 우선 → (미지 코드) 서버 message fallback → errorUnknown`. push/이메일은 발송 시점에 locale 이 필요하므로 디바이스 등록 시 locale 수집 후 서버측 per-locale 템플릿 선택.

**Tech Stack:** Spring Boot 멀티모듈(template-spring), Flutter + intl/arb(template-flutter, `S.supportedLocales = [en, ko]`), FCM(FcmPushAdapter), Resend(ResendEmailAdapter).

## Global Constraints

- `ErrorInfo` enum 기존 코드 번호 재배치 금지(추가만). `com.factory.*` 패키지 불변.
- wire 계약(`{data, error:{code,message,details}}`) breaking change 금지 — 필드 삭제/이름변경 없음.
- Mapper/Converter 클래스 신설 금지, Controller→Service→Repository, cross-domain 은 Port.
- 커밋 Conventional Commits, **Co-Authored-By 트레일러 금지**(husky 차단). push 는 사이클 끝 일괄.
- flutter 계약 변경 시 `template-flutter/docs/api-contract/*` 동시 갱신(동기화 의무 — `error_code.dart` docstring).
- main 의 `docs/**` push 만 하면 `sync-docs.yml` 이 docs-template-spring 자동 sync — 별도 sync 사이클 불필요.

## 1. 핵심 결정 — 에러코드 기반 클라 번역 vs 서버측 Accept-Language

| 관점 | A. 코드 기반 클라 번역 (클라가 code→arb) | B. 서버 Accept-Language (MessageSource) |
|---|---|---|
| 서버 작업량 | **0** (계약 재정의만) | 95개 상수 + Bean Validation + GlobalExceptionHandler 를 MessageSource 로 전환, N앱 × M언어 번들 운영 |
| 클라 작업량 | 매핑 헬퍼 1개 + arb 키 ~30개 | 0 (단 Accept-Language 헤더 추가 필요) |
| 문구 수정 속도 | **스토어 심사 대기(느림)** — 신규/수정 문안은 앱 릴리스 필요 | 서버 배포로 즉시 수정 |
| push/이메일 적용 | 별도 해결 필요(어차피 클라 번역 불가 채널) | **동일하게 별도 해결 필요** — 비동기 발송엔 헤더가 없어 locale 영속화 필수. B 를 골라도 이 문제는 안 풀림 |
| 기존 자산 정합 | **error-codes.md 가 이미 "code 가 계약, Flutter 가 매핑" 선언**. arb 파이프라인·번역 워크플로 존재 | 서버에 두 번째 번역 파이프라인 신설(1인 운영에 이중 부담) |
| 오프라인 에러 | NETWORK/TIMEOUT 은 어차피 클라 번역 — **단일 경로로 통일** | 오프라인 에러만 클라, 서버 에러만 서버 — 경로 이원화 |
| 언어 추가 | arb 파일 1개 추가(앱별) | 앱별 × 서버 번들 동시 추가 |

**권고안: A + 서버 message fallback (hybrid).**

- 계약: **`code` 만 유저 대면 계약. `message` 는 (1) 디버그/로그용 (2) 클라가 모르는 코드에 대한 표시 fallback.** wire 무변경.
- 표시 규칙(클라): `code→arb 매핑 → (미지 코드) 서버 message → errorUnknown`. 현재 `login_screen.dart` 관행(`errorMessage ?? _resolveErrorCode`)의 **우선순위 반전**.
- 1인 운영·앱 공장 trade-off 반영: 스토어 심사가 느린 약점은 **fallback 경로가 흡수** — 서버가 새 코드를 먼저 배포해도 구버전 앱은 서버 message(즉시 수정 가능)를 보여주므로 UX 공백이 없고, 다음 앱 릴리스에서 arb 매핑으로 승격하면 된다. B 의 유일한 강점(즉시 수정)을 fallback 으로만 취하고, B 의 비용(서버 번역 번들 상시 운영)은 지불하지 않는다.
- 따라서 서버 enum 의 한국어 message 는 **당분간 그대로 둔다**(fallback 용으로 유효). 앱별 주력 시장이 영어면 파생 레포에서 message 만 영어로 교체해도 계약이 안 깨진다 — 이것이 "서버 문구는 즉시 수정 가능" 특성의 활용처.

### 채널별 전략

| 채널 | 번역 주체 | 방식 |
|---|---|---|
| API 에러 메시지 | **클라** | code→arb 매핑, 미지 코드는 서버 message fallback (Task 2) |
| push 문안 | **서버** | 클라 번역 불가(비동기·OS 알림). 디바이스 locale 수집 → per-locale Template 선택, 미수집 시 앱별 기본 locale (Task 3) |
| 이메일 (인증·재설정) | **서버** | 동기 트리거 → 요청 `Accept-Language` 사용, 없으면 앱별 기본 locale (Task 3) |
| 이메일 (billing 등 비동기) | **서버** | push 와 동일 — 유저의 최근 디바이스 locale (Task 3) |
| 게시물·admin 발송 문안 | **운영자** | 번역 대상 아님 — 운영자가 타겟 시장 언어로 작성(앱 공장: 앱=단일 시장이 기본). 기계번역 out of scope |

**단계 분할**: Task 1(계약 문서화, 코드 무변경) → Task 2(flutter 매핑 헬퍼) → Task 3(push/email locale). Task 1·2 만으로 A 안이 완결되고, Task 3 은 독립 후속.

/// 표시 우선순위 (api-contract/error-codes.md §표시 규칙):
/// ① code→arb ② 서버 message (미지 코드 fallback) ③ errorUnknown.
String localizeApiError(String? code, String? serverMessage, S s) {
  final mapped = _mapCode(code, s);
  if (mapped != null) return mapped;
  if (serverMessage != null && serverMessage.isNotEmpty) return serverMessage;
  return s.errorUnknown;
}

String? _mapCode(String? code, S s) => switch (code) {
      ErrorCode.invalidCredentials => s.errInvalidCredentials,
      ErrorCode.refreshTokenExpired ||
      ErrorCode.refreshTokenInvalid => s.errSessionExpired,
      ErrorCode.socialAuthFailed => s.errSocialAuthFailed,
